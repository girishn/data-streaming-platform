# ADR-007: CFK Internal mTLS via cert-manager

## Decision
Use cert-manager with a self-signed ClusterIssuer to issue TLS certificates for CFK components within the EKS cluster. A platform CA (`cfk-ca`) is issued from the self-signed issuer, and a namespace-scoped `cfk-ca-issuer` issues component certificates from that CA.

## KB Source
`09-Security-Architecture/mtls-oauth.md` — CFK on EKS internal mTLS vs external connection model

## Rationale
The KB clearly separates two auth paths for CFK Connect workers:
- **Internal (within EKS)**: mTLS — CFK generates and manages certs for worker-to-worker and Connect REST API communication. cert-manager is the standard lifecycle manager.
- **External (→ Confluent Cloud)**: SASL/PLAIN over TLS — API key from Secrets Manager. mTLS client auth to Confluent Cloud requires Certificate Identity Pools (Dedicated cluster feature), which can be added later.

cert-manager `autoGeneratedCerts: true` on the CFK `Connect` CR delegates all certificate lifecycle to cert-manager, eliminating manual certificate rotation.

## Consequences
- cert-manager must be installed and the `cfk-ca-issuer` must exist before the `Connect` CR is applied.
- Internal CN principal names (`User:connect`) are determined by cert-manager certificate subjects — consistent across pod restarts.
- External Confluent Cloud connections are not affected by the internal CA.
